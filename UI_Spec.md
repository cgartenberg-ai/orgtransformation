## 2. Design Language: The Digital Herbarium

### Visual Concept

The interface should evoke a **modern naturalist's field guide** â€” clean, systematic, but with warmth and craft. Think: Muji meets Edward Tufte meets a beautifully designed botanical reference.

**Key Visual Principles:**

1. **Specimen Cards as Primary Objects**
   - Cards are the hero element â€” substantial, tactile-feeling
   - Subtle paper texture, slight shadows suggesting physicality
   - Expand smoothly to reveal layers of detail
   - Feel like something you'd want to collect

2. **Information Density Done Right**
   - Dense but not cluttered
   - Small multiples where appropriate
   - Progressive disclosure â€” overview first, details on demand

3. **Botanical Color Palette**
   - Primary: Deep forest green (#1B4332) â€” scholarly, grounded
   - Secondary: Warm cream (#FAF3E0) â€” paper, specimen cards
   - Accent: Amber (#D4A373) â€” highlighting, calls to action
   - Supporting: Sage (#84A98C), Charcoal (#2D3436)
   - Data visualization: Muted, natural tones (avoid neon)

4. **Typography**
   - Headlines: Serif with character (e.g., Fraunces, Libre Baskerville)
   - Body: Clean sans-serif (e.g., Inter, Source Sans)
   - Data/Labels: Monospace for taxonomic codes, metrics
   - Generous line height, comfortable reading

5. **Iconography**
   - Custom icons drawing on botanical illustration style
   - Structural models could have subtle visual signatures
   - Avoid generic tech iconography

### Motion Principles

- **Purposeful, not decorative** â€” animation should aid comprehension
- **Cards bloom open** â€” expansion feels organic, not mechanical
- **Timeline scrubbing** â€” smooth, responsive to input
- **Tension map** â€” specimens drift slightly, feel alive
- **Page transitions** â€” subtle crossfades, no jarring jumps

## 6. Recommended Tech Stack

### Frontend Framework: **Next.js 14 (App Router)**

**Why:**
- Server-side rendering for SEO (academics find via Google)
- Static generation for specimen pages (fast, cacheable)
- API routes for research tool integration
- React ecosystem maturity
- Excellent TypeScript support

### Styling: **Tailwind CSS + shadcn/ui**

**Why:**
- Rapid development with utility classes
- shadcn/ui provides accessible, customizable components
- Easy to maintain botanical color palette as CSS variables
- Good dark mode support (for late-night academics)

### Animation & Graphics: **Framer Motion + D3.js**

**Framer Motion** for:
- Card animations (expand, hover, transitions)
- Page transitions
- Micro-interactions
- List animations (specimens appearing/filtering)

**D3.js** for:
- Tension Map (2D scatter plot with force simulation)
- Evolution Timeline (custom timeline visualization)
- Taxonomy matrix (interactive heatmap)
- Any data-driven visualizations

**Why this combination:**
- Framer Motion handles UI animation elegantly in React
- D3.js is the gold standard for custom data visualization
- D3's force simulation perfect for the "specimens gently floating" effect
- Both are well-documented, production-ready

### Optional: **React Flow** for Evolution Timelines

If we want node-based timeline visualization with connections, React Flow provides:
- Draggable nodes
- Connections between layers
- Pan and zoom
- Could visualize how organizations branch/evolve

### Data Layer: **MDX + JSON + Supabase**

**Specimen Content: MDX files**
- Rich content with embedded components
- Version controlled in git
- Easy to edit, review, diff
- Can include React components inline

**Structured Data: JSON files (initially)**
- Taxonomy metadata
- Classification data
- Source registry
- Simple to start, migrate to DB later if needed

**User Data & Search: Supabase**
- User accounts (My Herbarium)
- Saved collections, notes
- Full-text search across specimens
- Real-time updates for research status

**Why this hybrid:**
- Start simple (files), scale as needed (database)
- Content lives close to code (easy to update)
- User data properly isolated
- Supabase provides auth + database + search in one

### Content Management (Optional): **Sanity or Keystatic**

If non-technical users need to edit specimens:
- Sanity: Full headless CMS, real-time collaboration
- Keystatic: Git-based, simpler, lives with code

For now, recommend starting with MDX files and adding CMS later if needed.

### Deployment: **Vercel**

**Why:**
- Native Next.js support
- Edge functions for API routes
- Preview deployments for content review
- Simple, reliable, fast

### Search: **Algolia or Supabase Full-Text**

For specimen search:
- Algolia: Best-in-class search UX, faceted filtering
- Supabase: Good enough for start, already in stack

Recommend starting with Supabase, upgrade to Algolia if search becomes critical path.

## 7. Graphics Engine: Making Cards Come Alive

### Animation Philosophy

The goal is **subtle life** â€” specimens feel like living documentation, not static records. Think: gentle breathing, not dancing.

### Specific Effects

**1. Specimen Cards**

```javascript
// Framer Motion variants for card
const cardVariants = {
  rest: {
    scale: 1,
    boxShadow: "0 2px 8px rgba(0,0,0,0.1)",
    y: 0
  },
  hover: {
    scale: 1.02,
    boxShadow: "0 8px 24px rgba(0,0,0,0.15)",
    y: -4,
    transition: { duration: 0.2, ease: "easeOut" }
  },
  tap: { scale: 0.98 },
  expanded: {
    // Smooth height animation handled by AnimatePresence
  }
}
```

**2. Tension Map (D3 Force Simulation)**

```javascript
// D3 force simulation for specimen positioning
const simulation = d3.forceSimulation(specimens)
  .force("x", d3.forceX(d => tensionScale(d.tensionScore)).strength(0.8))
  .force("y", d3.forceY(height / 2).strength(0.1))
  .force("collide", d3.forceCollide(30))
  .alphaDecay(0.02)  // Slow decay = gentle continued motion
  .on("tick", updatePositions);
```

**3. Evolution Timeline**

```javascript
// Timeline with layered reveal
const timelineVariants = {
  hidden: { opacity: 0, x: -20 },
  visible: (i) => ({
    opacity: 1,
    x: 0,
    transition: { delay: i * 0.1, duration: 0.3 }
  })
}
```

**4. Filter Transitions (List)**

```javascript
// AnimatePresence for filtering specimens
<AnimatePresence>
  {filteredSpecimens.map(specimen => (
    <motion.div
      key={specimen.id}
      initial={{ opacity: 0, height: 0 }}
      animate={{ opacity: 1, height: "auto" }}
      exit={{ opacity: 0, height: 0 }}
      layout  // Smooth reordering
    >
      <SpecimenCard {...specimen} />
    </motion.div>
  ))}
</AnimatePresence>
```

**5. Page Transitions**

```javascript
// Subtle crossfade between pages
const pageTransition = {
  initial: { opacity: 0 },
  animate: { opacity: 1, transition: { duration: 0.3 } },
  exit: { opacity: 0, transition: { duration: 0.2 } }
}
```

### Performance Considerations

- Use `will-change` sparingly
- Prefer `transform` and `opacity` for animations (GPU-accelerated)
- Lazy load specimen cards below fold
- Virtualize long lists (react-window)
- D3 simulations should stop when not visible

## 8. API Structure

### Specimens API

```
GET  /api/specimens                    # List all (with filters)
GET  /api/specimens/[id]               # Get single specimen
GET  /api/specimens/[id]/evolution     # Get evolution layers
POST /api/specimens/[id]/save          # Save to My Herbarium
```

### Search & Filter API

```
GET  /api/search?q=...                 # Full-text search
GET  /api/filter?model=1&orientation=structural&industry=pharma
GET  /api/match?regulatory=high&talent=constrained&...  # Situation matcher
```

### Research API

```
POST /api/research/start               # Trigger research cycle
GET  /api/research/status              # Get current status
GET  /api/research/history             # Past runs
```

### User API (Authenticated)

```
GET  /api/me/collections               # User's collections
POST /api/me/collections               # Create collection
POST /api/me/notes                     # Add note
GET  /api/me/recent                    # Recently viewed
```

## 11. Open Design Questions

1. **Authentication Model**: Open access vs. freemium vs. institutional licenses? (Affects My Herbarium scope)

2. **User Contributions**: Should users be able to suggest specimens or edits? (Community curation vs. quality control)

3. **Notification System**: Email digests for followed specimens/topics? (Engagement vs. spam)

4. **Mobile Experience**: Full parity or focused subset? (Tension Map may need rethinking for mobile)

5. **Embedding**: Should specimens be embeddable in other sites? (Academic blogs, internal wikis)

6. **API Access**: Public API for researchers? (Data portability vs. value capture)
